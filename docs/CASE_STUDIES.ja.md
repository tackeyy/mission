# ケーススタディ: レビューゲートが実際に捕まえたもの

データ取得日: 2026-07-07。

## 出所と限界

- 出所はメンテナのローカル private リポジトリ群にある本番 `.mission-state`
  ファイルです。state ファイル自体は**公開しません**。プロジェクト名は以下で
  匿名化しています。このリポジトリ単体からは再現できないため、benchmark では
  なく開示された運用証跡として読んでください。
- スコアは mission 自身の reviewer composite（1–5 スケール、合格ゲート 4.0）
  です。評価対象と同じシステムの内部で生成されています。`push-score` は
  水増しされた自己申告スコアを拒否しますが、それでも独立した blind judging
  ではありません。
- このページは**比較主張**（「plain single pass より良い」「`/goal` より良い」）
  を一切支持しません。paired comparison は
  [`benchmarks/mission-vs-goal/README.ja.md`](../benchmarks/mission-vs-goal/README.ja.md)
  を参照してください。

## 全体像

**451 件の scored 本番ミッション**（score history を持つ unique state、
重複排除済み）の集計:

| 項目 | 値 | 読み方 |
|---|---:|---|
| iteration 1 で 4.0 ゲートを通過 | 427 / 451 (95%) | 大半の作業でループは素通し。レビューコストを足すだけで何も変えない。 |
| iteration 1 で 4.0 未満 | 24 (5%) | ゲートが効いたテール。これらは ship せずに反復を強制された。 |
| multi-iteration ミッション | 44 | ゲート強制の再実行と、4.0 以上でも findings 起点の再実行を含む。 |
| — 反復で composite が改善 | 27 | 計測された改善。例: 2.80 → 4.20、3.10 → 4.60、0.96 → 4.80。 |
| — 反復で composite が不変 | 15 | 正直なネガティブ: スコアを動かさずコストだけ足した反復。 |
| 人間の承認・入力待ちで halt | 7 | 不可逆操作（本番 DB migration、security audit の公開、本番 API cap 変更）を明示承認まで停止。 |

平均より分布が重要です。レビューゲートがコストに見合うのは**少数のテール**
であって、平均を押し上げることではありません。タスクがすべて 95% 側なら、
単発の丁寧な 1 pass の方が同じ成果物を速く得られます。

## 代表ケース

プロジェクトは匿名化しています: **A** = operations bot、**B** = SNS 運用
SaaS、**C** = 小規模公開 Web サービス、**D** = 本リポジトリ自身のツール。
以下の findings は iteration 1 の記録済み reviewer 出力と iteration 2 以降の
記録済み差分の要約です。

### Case 1 — 誤った根本原因分析が ship 寸前だった（A: 2.80 → 4.04 → 4.20）

エラー増加調査がエラー率 51.9% と報告。レビューで数え直すと 79.4%（mock
エントリが分母に混入）でした。さらに初回 pass は自分が原因とみなした既存
コード構造を誤読しており（提案したガードは既に存在、真因は複合コマンドを
分割せず一括拒否する composition check）、成立しないサンドボックス特性を
根拠に修正を正当化していました。iteration 2 で率・機構・根拠を修正、
iteration 3 でテスト付き実装計画を追加。ゲートがなければ、事実誤認 3 件を
含む根本原因報告がそのまま修正の前提になっていました。

### Case 2 — テストは緑、レイアウトは崩壊（B: 3.10 → 4.60）

UI 変更は初回 pass で型チェックとフルテストスイートを通過。レビューが
サーバーレンダリングされた固定幅とクライアントのトグル state の乖離
（トグル時に 156px の視覚的な隙間）、accessibility 属性の欠落、未テストの
state パス 2 件を検出。iteration 2 で全修正。静的チェックだけなら見えない
レイアウトバグが ship されていました。

### Case 3 — 存在しないコンポーネントを前提にした設計書（B: 3.70 → 4.70）

ナビゲーション移行設計が、UI ライブラリに存在しない icon コンポーネントを
前提にし、未確認の外部参照を複数、事実として断定していました。実装の最初の
ステップでコンパイルが失敗する状態です。iteration 2 で（grep で確認の上）
コンポーネント選定を差し替え、未確認の断定をすべて「確認済みに直すか削除」
しました。

### Case 4 — ツールチェーンに見えない実行時 UI バグ（B: 3.60 → 4.40）

サイドバー実装は `tsc` と 1,200 件超の unit test を通過しながら、閉じる
ボタンの二重配置、ナビリンクの欠落、ドロワーを壊す event propagation、
モバイルのトグル不具合を含んでいました。5 件すべてが reviewer findings で、
iteration 2 で修正されました。

### Case 5 — 緑のテストの裏のセキュリティ関連ギャップ（C: 3.40 → 4.65）

認証・ダッシュボードの作業計画で、識別子の正規表現が API・middleware・DB
の各層で食い違い、ログインリンクの期限検証がありませんでした。どちらも
セキュリティに関わり、どちらも初回 pass の緑のテストを生き延びていました。
iteration 2 で High 1 件と Medium/Low 13 件を修正し、最終 pass で期限切れ
（401）・重複（409）・不正（400）の各パスを明示的に検証しました。

### Case 6 — ゲートは reviewer のノイズも濾す（D: 3.87 → 4.50）

本ツール自身の skill アーキテクチャ再設計分析が、「今日着手」項目に
ファイル名・関数名のポインタを欠いたため usability を落としました —
書かれたままでは着手不能です。iteration 2 で具体的ポインタを追加。特筆
すべきは、reviewer finding の 1 件自体が誤りで、orchestrator が反復前に
grep 証跡で棄却したことです。ループは双方向に adversarial であり、それが
false findings による反復回数の水増しを防いでいます。

### Halt Case — 不可逆操作を承認なしに完了扱いしない（B: 3.78、halt）

4 つの issue を実装するミッションが、close せずに自分から停止しました。
本番データベース migration（物理 drop）の適用と live 投稿検証が、明示
確認を要する不可逆操作として flag されたためです。halt 理由は state に
そのまま記録されており、承認後の後続セッションが 4.50 で完遂しました。
CI が緑なら完了に向かう completion-driven な single pass には、ここで
「完了を拒否する」機構がありません。

## この証跡が支持すること・しないこと

支持すること:

- レビューゲートは、本番ミッションの約 5% で実在の検証可能な欠陥（事実
  誤認、実行時 UI バグ、セキュリティ関連の不整合、着手不能な計画）を捕まえ、
  それらの成果物を反復で計測可能に改善する。
- halt ゲートは不可逆な本番操作を人間の承認まで停止する。これは品質スコア
  から独立したガバナンス特性である。

支持しないこと:

- mission が plain single pass より平均品質を上げるという主張。これまでに
  完了した paired benchmark run は completion / validator 指標で同点、
  mission 側の時間・コストが高い
  （[`benchmarks/mission-vs-goal/report.ja.md`](../benchmarks/mission-vs-goal/report.ja.md) 参照）。
- 5% の catch rate が他ユーザーのタスク構成に移転するという主張。
- スコア不変の反復 15 件は、品質変化の計測なしに記録されたコストであり、
  その削減は未着手の課題。
