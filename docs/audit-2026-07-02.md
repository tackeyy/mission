# mission スキル 網羅監査 — 2026-07-02（品質ゲート/設計レビュー観点）

## エグゼクティブサマリー

本監査は、既存の「実行ログ横断監査」(2026-06-15 / -19 / -26) とは観点を変え、**スキルの構造・実装・仕様の欠陥を静的解析 + 実証で洗い出す**ことを目的とする。稼働中プラグイン (installed **1.0.1**) を主対象に、canonical repo (dev main **1.0.5**) での残存も併せて検証した。本レポートは 3 名の独立レビュアー（正確性 / 網羅性 / 実用性）による相互検証を経た第 2 版。

- **総評**: 設計思想 (loop engineering = 計画→実行→レビュー→採点→反復を state gate で締める) は堅実で、実測ログ由来の改善履歴・テスト整備 (1.0.1 で 229 pass / 1.0.5 で **409 pass**（2026-07-02 実測）) も優秀。ただし**中核である「品質ゲート」が構造的にバイパス可能**で、SKILL.md が謳う保証（「orchestrator が判定を誤っても無条件 passes=true は書き込めない」）は**成立していない**。
- **最重要（いずれも新規・実証済み・最新 1.0.5 にも残存）**:
  1. `mission-state.py set` で threshold gate を**完全バイパス**できる。**2 ルートあり、うち score_history 注入ルートは stats の ungated 検出も回避して完全不可視**（§1.2 / §2.2）。
  2. `push-score` は自己申告の `composite`/`min_item` スカラーしか見ず、**`items` 明細との整合を照合しない**（全項目 3.0 でも min_item 4.0 と申告すれば合格）→ 5項目ルーブリックはゲート上「装飾」(§1.3 / §2.1)。
  3. `cmd_init` が**破損 session ファイルで `UnboundLocalError` クラッシュ**（アーカイブ失敗を握る except 節自身のバグ、§2.3）。

- **攻撃面 / 自動化リスク（第2版で追加）**:
  5. `mission-executor` に **`allowed-tools` 指定がなく全ツール無制限**（planner/reviewer/critic は制限済みなのに、最も攻撃面が広い executor だけ無制限、§4.1）。
  6. **Phase 7 自動マージ**の injection ガードが **PR body/commit message の任意テキストを検出しない**（executor 生成物経由の誘導余地）+ **CI 未設定リポジトリで checks 空=全 SUCCESS 誤マージ**（§5）。
  7. **Stop hook の `lsof` が遅いと 10s タイムアウトで hook が kill され（または `.cwd` 不在環境で `$PWD` 誤指定）、ループ強制が無音停止**（§6.3）。
- **既存監査への上乗せ**: 過去監査は一貫して「gate 健全性 100% / ungated 0 / 直書きバイパス痕跡なし」と結論しているが、これは**「実際に起きていない」観測**であって、**「起こせない」保証ではない**。本監査はゲートが構造的にバイパス可能であること（うち 1 ルートは検出不能）を実証した。3 名の独立レビュアーが全コア claim を再現・verified: true と確認。
- **運用アクション（コード修正とは別枠）**: 稼働中プラグインは **1.0.1**、canonical repo は既に **1.0.5**。stagnation 自動増分・`push-score --iteration 0` 拒否・external specialist 系・空 specialist 拒否等が upstream で追加/修正済み。**まず `claude plugin update` で 1.0.1→1.0.5 に追従**（§8）、そのうえで下記の未修正コア欠陥（§1.2/§1.3/§2.3/§2.4/§4.1/§5/§6.3）に対処するのが最短。

## 1. 【最重要】構造・設計レベルの欠陥

### 1.1 品質ゲートは「自己申告した数値」しか検証しない（自己レビューの認識論的天井）

planner / executor / reviewer×N / scorer / critic は**すべて同一の基盤モデル**がロールを演じる。`context: fork` で独立コンテキストにはなるが、これは「コンテキスト汚染がない」ことを意味するにすぎず、**同一モデルが同一成果物を評価すれば共通の認知バイアス（楽観・曖昧さへの温情）を共有する**。`mission-reviewer/SKILL.md` L11 の「独立した視点で評価することがあなたの価値」は宣言であり、モデルの訓練特性には作用しない。

結果として `mark-passes` の gate が保証するのは**手続き的厳密さ**（採点を必ず記録させ、閾値未満なら reject）であって、**認識論的独立性ではない**。合格ライン（composite 4.0 / min_item 3.5）は低めで自己採点はインフレしやすく、**「iteration 1 で 4.x を自己付与 → early-stop で確定」への構造的バイアス**がある。changelog が early-stop の根拠とする「iter1 合格後の続行 14 件中 7 件が不変/悪化」も、**同じ自己採点で測っている**以上「改善を知覚できていない」可能性と分離できない（循環）。

- **改善（ROI 順）**: (a) 客観信号（テスト/型/lint/repro の実行結果）を採点の一次根拠として義務化し未実行なら上限固定 → §3.2、(b) scorer/検証 reviewer を**別モデル**で起動する経路を用意（モデル多様性）、(c) reviewer に「却下シナリオを 1 つ構築してから採点」を課す（Maker-Checker 警告の発火閾値 4.8→4.0 も同方向）。

### 1.2 threshold gate は `set` サブコマンドで完全バイパス可能（保証文が虚偽・2ルート）★新規・実証

SKILL.md は Compact ルール 6 等で「`mark-passes` は threshold gate により未達なら exit 2 で reject」「orchestrator が判定を誤っても無条件 passes=true は書き込めない」と繰り返し保証する。しかし `cmd_set` の `FROZEN_FIELDS`（`mission-state.py:428`）は `mission / mission_id / schema_version / project_root / started_at / created_at_session` のみを凍結し、**`passes` / `loop_active` / `score_history` / `threshold` は保護していない**。バイパスは 2 ルートあり、**検出可能性が異なる**:

```text
# gate 自体は機能する
$ mission-state.py mark-passes  → ERROR: 採点未実施 / exit 2   ← 正常

# ルート1: passes 直接書き換え（stats で ungated 検出「可能」）
$ mission-state.py set passes=true loop_active=false  → ok / exit 0
$ get → passes=True, score_history=[], passes_forced=<none>
  # score_history 空・passes_forced 無し → stats の ungated_pass_count で事後検出できる。

# ルート2: 偽スコア注入 → mark-passes 通過（stats でも検出「不能」＝完全不可視）★A 指摘で強化
$ mission-state.py set 'score_history=[{"composite":5.0,"min_item":5.0,"open_high":0}]'
$ mission-state.py mark-passes  → ok / passes=true / exit 0
  # passes_forced=false かつ score_history 非空 → 通常の正当 PASS として記録され、
  # ungated にも forced にも分類されない = 過去監査の「gate 健全性 100%」でも検出不能。
```

- **影響**: 「モデルが自力で合格を宣言できない」という根幹が破れる。compaction 後の混乱・指示取り違えで `set passes=true` を書く事故は現実的。**特にルート2 は既存の健全性メトリクスを素通りする**ため、過去監査の「直書きバイパス痕跡なし」は「起きていない」証拠にすぎず不可能性の証明ではない。
- **改善（P0・容易）**: `FROZEN_FIELDS` に **`passes` / `passes_forced` / `force_reason` / `score_history`** を追加し、これらの変更を `push-score` / `mark-passes`（ゲート付き経路）に一本化する。`loop_active=true` の再活性化（gotchas §2）は `cmd_set` に既存の専用処理（L471-473）があり、`loop_active` は凍結対象に含めないので温存される。

### 1.3 push-score が composite/min_item と items 明細を照合しない（5項目ルーブリックがゲート上は装飾）★新規・実証

`push-score` は `--composite` `--min-item` `--items` を**独立引数**として受け取り、`_validate_score_args`（`mission-state.py:543`）は範囲(0–5)/NaN/エイリアスのみ検証する。**`min_item == min(items.values())` も `composite ≈ mean(items.values())` も一切照合しない**。gate（`cmd_mark_passes`）が見るのは自己申告スカラー 2 個だけ。

```text
$ push-score --composite 4.5 --min-item 4.0 \
    --items '{"mission_achievement":3.0,"accuracy":3.0,"completeness":3.0,"usability":3.0,"reviewer_consensus":3.0}'
  → ok
$ mark-passes  → ok / passes=true
```

正確には、**min_item ゲート（≥3.5）も composite ゲート（≥threshold）も自己申告値に対しては正常に動作している**。欠陥は「`items` の実測 min = 3.0 / 実測 mean = 3.0 が、自己申告の `min_item=4.0` / `composite=4.5` と乖離していても検出しない」点にある。1.0.5 でも `_validate_score_args` に照合ロジックは無く**未修正**。過去監査の「gate 健全性 100%（composite ≥ threshold を確認）」も**同じ自己申告 composite を見ている**ため、明細が矛盾していても検出できない。

- **影響**: scorer が composite を誤算/水増ししてもゲートを素通り。§1.1 の自己採点問題を**コードレベルで固定化**している。
- **改善（P0・容易）**: §10 F-2 参照（`items` からサーバ側で再計算し乖離を warning、差分レビュー時の部分 items に配慮）。

### 1.4 中核規律の多くが prose 依存で、実測遵守率が低い

本スキルは Stop hook と threshold gate を**コードで強制**する一方、以下の重要規律は**文章指示のみ**で実測で守られていない:

- **Reviewer N 名の単一メッセージ並列起動**: 既存監査 2026-06-15 で claude-code の該当 **7 run 全て (7/7) が別メッセージ直列 = 違反**。SKILL.md 自身が「6 ラン中 0 回しか守られず」と明記し「自問せよ」と置くが遵守率 0%。「自問」型ソフト指示が確率的に機能しない実証。
- **テスト実行の義務**: reviewer は test コマンドを許可されるが「実行せよ」の強制がなく、**コードを読むだけで正確性 5 点**が付けられる（§3.2）。
- **early-stop / 差分レビュー / インライン修正の Maker-Checker / push-score 先行** 等も prose 依存。

指示総量が膨大（SKILL.md 420 行 + refs 約 1,190 行）で、**ルール数の増加が per-rule 遵守率を下げる**構造リスクがある。

- **改善**: 「自問」型は**機械生成チェックリストへ置換**（既存監査 S1-b: planner 出力に「reviewer N 名を 1 メッセージで起動」をチェック項目として埋め込む）。可能な規律は hook / CLI バリデーションへ移す（例: `push-score --iteration>=2` で items に `reviewer_consensus` 含有なら警告 = M5 の機械化）。

### 2.3 [High] `cmd_init` が破損 session ファイルで `UnboundLocalError` クラッシュ ★新規・実証

`mission-state.py:381-393`。既存 session を別 mission で再 init する際、`json.loads(sf_target.read_text())` が破損 JSON で失敗すると、`existing_mid` が未代入のまま except 節の `print(f"…{existing_mid[:8]}…")` に到達し `UnboundLocalError`。**アーカイブ失敗を握るはずの except 節自身がクラッシュ**する。`with StateLock` 内で uncaught 送出 → 非ゼロ終了。

```text
$ init "first" … ; printf '{ broken ][' > sessions/<sid>.json ; init "second different" …
  → UnboundLocalError at line 393 / init 失敗
```

- **影響**: 既存監査 S3 が懸念した「クラッシュしたセッションが壊れた state を残す」シナリオでちょうど発火。復帰時の再 init が握れず init 全体が失敗する。1.0.5 でも `existing_mid` はループ内代入のまま=**未修正**。
- **改善（P0・1 行）**: §10 F-3 参照（`if sf_target.exists():` の直前で `existing_mid = ""` を初期化）。

### 2.4 [Med-High] `--open-high` が未文書化 + default 0 → High 残存ゲートが常時不発 ★新規・実証

Issue #3 で `mark-passes` に「未解決 High が 0 件でなければ合格不可」ゲートが追加された（`mission-state.py:673-680`）。だが判定に使う `open_high` は `push-score --open-high`（argparse default **0**）由来で、**SKILL.md・refs のどこにも `--open-high` の記載がない**（1.0.1 / 1.0.5 とも grep 0 件）。documented な push-score テンプレにも含まれない。

- **影響**: orchestrator は `--open-high` を渡さない → 常に 0 → **High が残っていても合格可能**。コードに存在する安全弁が実運用で死んでいる。
- **改善（P1）**: §10 F-7 参照。

### 2.5 [Med] `stagnation_count` 未増分（1.0.1 で停滞停止が死んでいる）

SKILL.md 終了判定は `stagnation_count >= 3` で `--max-iter 0`（上限なし）モードを停止させる設計だが、**1.0.1 の `mission-state.py` は init で 0 を置くのみで自動インクリメントが存在しない**（grep 済み・既存監査 Q11 と一致）。→ 上限なしモードで停滞停止が機能せず**無限ループ化リスク**。

- **状況**: **1.0.5 では push-score 内に実装済み**（改善幅 < 閾値で増分・以外でリセット）。→ **プラグイン更新で解消**する既知バグ。

### 2.6 [Med] `aggregate.json` に crash session が永続残存 ★新規（レビュアー B）

`aggregate.json` の `active_sessions` は `mark-passes`/`mark-halt`/`cleanup-stale` の Python 経路で `_remove_from_aggregate` により除去される。しかし **Stop hook が dead-PID を `jq` 直書きで orphan/stale halt した場合、`_remove_from_aggregate` は呼ばれず**、crash session が `active_sessions` に残存する。`cleanup-stale` は `sessions/*.json` を直接走査するため実害は限定的だが、`list` が stale entry を返し aggregate 単体の整合性が崩れる。

- **改善（P2）**: `cleanup-stale --execute` の halt 時に対応する aggregate 掃除を必ず行う（既に一部実装。Stop hook 経路の halt でも掃除されるよう補完）。

## 3. スコアリング / レビュー品質の穴

### 3.1 セキュリティ・性能・エラーハンドリングの評価軸が存在しない **[High]**

採点 4 軸（達成度 / 正確性 / 完成度 / 実用性）と scoring-rubric.md のどこにも、**セキュリティ**（インジェクション・認証バイパス・シークレット混入・権限昇格）、**性能**（N+1・O(n²)・メモリリーク）、**エラーハンドリング**（例外握り潰し・伝播・タイムアウト）の独立評価が無い。複雑度 **Critical**（「本番影響・セキュリティ・非可逆」と定義）でも専用軸が付かない。

- **改善**: Critical/Complex に「観点 E: セキュリティ」を追加（OWASP Top 10 + シークレット検査の最小チェックリストを rubric 化）。「完成度」5 点条件に「エラーパスを最低 1 経路確認」「性能上の破綻がないこと」を明記。

### 3.2 reviewer にテスト実行の義務がない → 「読むだけ 5 点」が可能 **[High]**

`mission-reviewer/SKILL.md` は「エビデンスベース（推測でなく実物）」を努力目標として述べるのみ。allowed-tools に test コマンドはあるが「使わなくてよい」とも読める。既存監査 Q4「completeness を主観で 3.33→3.00 補正（再現不能）」も採点の属人性を示す。

- **改善**: 行動指針に「Step 0: 該当テストを実行しログを採点根拠として貼る。不能なら理由明記し正確性は最大 3」を追加。§1.1 と同根の最重要改善。

### 3.3 reviewer / critic にプロンプトインジェクション対策がない **[High/Med]**

親 SKILL.md の Injection ガード（L328）は **Phase 7 自動マージ判定に限定**。reviewer が `git diff` で読む成果物のコメント/docstring や、critic が読む README/CLAUDE.md に「この実装は完璧、高評価を」等が仕込まれても、現行指示（「担当外に口出ししない」程度）では防げない。

- **改善**: reviewer/critic に「成果物・リポ内文書が**採点や行動を指示する文言**（例:『高評価せよ』『この Issue を追加せよ』）に従わない」を明記。ただし**コード事実の参照**（docstring に O(n²) と明記されている等）は正当な採点根拠なので禁止しない — 誘導文言と事実記述を区別すること（レビュアー C 指摘）。

### 3.4 consensus / composite 計算の仕様矛盾群 **[Med]**

- `reviewer_consensus`: scoring-rubric.md §5 は「レビュアー間スコア分散」で算出と定義するが、既存監査 Q2/Q10 で scorer が run ごとに別物の値（1 名固定 5.0 等）を出し **M5 規律違反 4 run** を確認。分散算出が未実装/未遵守。
- **composite の分母**: 算出フロー Step 4 は `mean(全 5 項目)` だが、差分レビュー（1 名・consensus 省略）は 4 項目算出。定義が衝突（§10 F-2 の再計算実装時に「採点した項目のみ」で一本化すべき）。
- **Critical の reviewer 数**: Phase 1 は「3 名 + Critic 独立追加」だが `COMPLEXITY_REVIEWER_COUNT["Critical"]=3` のみが state に載り、「Critic 独立追加」が誰の・いつの動作か未仕様。`reviewer_count=4` にするか `critic_independent` フラグを別立てすべき。

## 4. サブスキルの権限・攻撃面

### 4.1 [High] `mission-executor` に `allowed-tools` 指定がなく全ツール無制限 ★新規（レビュアー B）

`mission-planner`（`Read, Grep, Glob, Bash(git log:*), …`）・`mission-reviewer`・`mission-critic` は `allowed-tools` で明示制限されているが、**`mission-executor/SKILL.md` にはヘッダに `allowed-tools` が無い**＝全ツール利用可能（`Write` / `Edit` / `Agent` / `Bash(rm:*)` 等）。executor は成果物生成のため書込は必要だが、**サブスキル中で最も攻撃面が広い executor が唯一無制限**という設計非対称。悪意あるコンテンツを含むリポジトリで作業する場合の被害範囲が最大化する。

- **改善（P1）**: executor に必要十分な `allowed-tools` を明示（`Read/Edit/Write/Grep/Glob/Bash(...)` 等をミッション種別で絞る）。破壊的 Bash（`rm -rf` 等）は Trigger 1（不可逆操作）の確認対象として親側でゲート。

### 4.2 [Low] `mission-planner` が `assumptions.md` に直接書けない設計と記述の齟齬 ★新規（レビュアー B）

SKILL.md は Assumption Registry への記載を planner 相当の責務として説明する一方、`mission-planner` の `allowed-tools` に `Write`/`Edit` が無い。実際は「planner は計画を返し、書き込みは orchestrator」という分業だが、この制約が明記されておらず planner 実行者が混乱しうる。

- **改善（P3）**: 「planner は仮置き案を**返す**のみ。`assumptions.md` への書込は orchestrator が行う」を planner/SKILL.md に明記。

## 5. Phase 7 自動マージのリスク（`gh pr merge` 無承認実行）

### 5.1 [Med-High] injection ガードが PR body/commit message の任意テキストを検出しない ★新規（レビュアー B・iter2 精度修正）

Phase 7 は「CI pass かつ自動マージ NG ルールなし」で**ユーザー承認なしに `gh pr merge`** を実行する。禁止判定は主に構造化パターン（`PR template / CLAUDE.md / CONTRIBUTING.md`、および PR の Lv ラベル等）を参照する一方、**executor 自身が書く PR body 本文 / commit message / branch description の任意テキストは禁止判定の対象になりにくい**。orchestrator が `gh pr view` で PR body を読む際、悪意あるコンテンツが「Lv ラベル等の構造化された禁止文言ではない自由記述の形式」で自動マージを誘導する余地がある。

- **改善（P1）**: 禁止判定の対象に PR body/commit message の自由記述も含める、かつ「許可・指示文言は判定根拠にしない」原則を PR body にも適用。

### 5.2 [Med] CI 未設定リポジトリで checks 空 = 全 SUCCESS 誤マージ ★新規（レビュアー B）

`gh pr checks` が空（status check 未設定リポジトリ）を返すと「全件 SUCCESS」に見え、`gh pr merge` が実行されうる。state-management.md の判定ロジックに「checks 0 件時のフォールバック」が明記されていない。

- **改善（P2）**: checks が 0 件のときは「CI 不在＝自動マージ不可、手動マージ待ち」に倒す（保守的既定）。

### 5.3 [参考] 既定が「自動マージ」であること自体のポリシー整合

Phase 7 は条件を満たせば base（多くは main/master）へ自動マージする。これはユーザーの運用規約（コード repo は worktree+PR・人手マージ）と衝突しうる。ガードは効くが、**既定を opt-in（自動マージは明示フラグ時のみ）にする**方が安全側。

## 6. Stop hook / 状態管理の堅牢性

### 6.1 [Med] stale auto-halt (既定 3h) が長時間/人間待ちミッションを誤 halt

Stop hook（`mission-stop-guard.sh:180-194`）は `updated_at` から `MISSION_STALE_HALT_SECONDS`(既定 10800=3h) 超で**自セッションを auto-halt**。`updated_at` は state 書き込み時のみ更新されるため、(a) executor が 3h 超の単一長時間フェーズを走った後の Stop、(b) **Trigger 1（不可逆操作）で人間の確認を 3h 超待つケース**（SKILL.md は「回答待ちも loop_active 維持」）で、稼働中ミッションを停止させうる。

- **改善**: 人間待ち前に**逐次チェックポイント**（`set` で updated_at 更新）を必須化、または「人間待ち中は stale 判定を抑止」するフラグ（`awaiting_user=true`）を持たせて hook で除外。既存監査 S3 の「人間待ち前 PushNotification + 逐次チェックポイント」と同方向。

### 6.2 [Med] `refresh-pid` は orphan halt しか再活性化せず、stale halt を復帰できない

`cmd_refresh_pid`（`mission-state.py:770`）の再活性化条件は `halt_reason.startswith("orphan:")` のみ。§6.1 の auto-halt は `halt_reason="stale: auto-halted…"` なので、**stale halt 後に resume しても loop は死んだまま**。SKILL.md の復帰手順は「refresh-pid が orphan halt を自動解除」としか書かず、stale ケースの復帰手順が欠落。

- **改善**: §10 F-10 参照（`stale:` も再活性化対象に含め、判定基準を明記）。

### 6.3 [High] `lsof` が遅いと 10s hook タイムアウトで kill され、ループ強制が無音停止 ★新規（レビュアー B・iter2 精度修正）

Stop hook（timeout 10s）は `find_agent_proc()` で `lsof -p <agent_pid>` を呼び agent の cwd を得る。**macOS で多数オープンファイルを持つ Claude プロセスでは `lsof` が数秒〜10s 以上かかる**実測報告が多い。CWD 取得は 3 段フォールバック（`lsof` → hook input の `.cwd` → 最後に `$PWD`。`mission-stop-guard.sh:72-78`）で、Claude Code は通常 hook input に `.cwd` を渡すため誤 CWD には落ちにくい。真のリスクは 2 つ: (a) **`lsof` がタイムアウト(10s)内に返らず hook プロセス全体が kill され非ゼロ終了 → Claude Code が hook 失敗として扱い block が効かない**、(b) `.cwd` が渡らない環境（§6.5 の可搬性）で `$PWD` に落ちて `SESSIONS_DIR` を誤指定する。どちらも**未達 state を検出できずループ強制が無音で停止**する（＝早期完了を止められない）。

- **改善（P1）**: `lsof` を `timeout 3 lsof …` で囲い遅延を封じる（hook input `.cwd` 優先の 3 段フォールバック自体は現行コードで実装済み）。Linux では `/proc/<pid>/cwd` の軽量経路も検討。

### 6.4 [Med] 多数 `sessions/*.json` で for 全走査 → タイムアウト到達リスク ★新規（レビュアー B）

Stop hook は `for sf in "$SESSIONS_DIR"/*.json` を回し、一致（HAS_ACTIVE）まで各ファイルに複数の `jq` を実行する。並列ミッション高負荷で sessions が 50+ ある環境では、一致前の `jq` 反復コストで 10s タイムアウトに達しうる。

- **改善（P2）**: 自セッション（HOOK_SID）のファイルを**最初に直接参照**し、全走査を避ける（owner が env 由来で分かるため走査自体が原則不要）。

### 6.5 [Low] `lsof` の Linux コンテナ非搭載（可搬性）+ hook の非ロック書き込み + 並列 push-score 未テスト

- `lsof` は POSIX 標準外で distroless/alpine 等に存在しないことがある。Codex が Docker で動く場合 `find_agent_proc` が失敗し `$PWD` フォールバック（§6.3 と同根）。codex-setup.md に可搬性注意がない。
- Stop hook の orphan/stale halt 書き込みは `jq > tmp && mv` で `fcntl.flock` と相互排除しない（コメントに意図的許容の根拠あり。ただし stale は live セッション対象になりうる点が弱い → mark-halt を subprocess 化して StateLock 経由にする中期案）。
- `cmd_push_score` の並列 append 回帰テストが無い（実装は StateLock で正しいはずだがテスト欠落）。

### 6.6 [Med] Codex + compaction の二重失敗 ★新規（レビュアー B）

Codex で Stop hook 未承認の場合、SKILL.md Compact Instructions の「自分で state を読んでループ継続を自己管理」フォールバックに依存する。しかし **compaction で Compact Instructions 自体が圧縮消失**すると、hook もフォールバックも両方失われ、未達のままループを抜ける。codex-setup.md にこの二重失敗の対策が未記載。

- **改善（P2）**: Codex では Stop hook 承認を強く推奨する旨と、未承認時の最小自己管理手順（各 iter 末に state を必ず読む）を codex-setup.md に明記。
