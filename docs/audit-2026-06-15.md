## 1. 速度(実行速度)の所見

### S1. 【最優先】Reviewer 並列ルール 7/7 違反 — 自己チェック指示が無効
claude-code で reviewer skill を使った **7 run 全て**が、N 名の reviewer を**別メッセージで直列起動**していた
(単一メッセージ複数 Skill = P4 ルール違反)。

| run | cx | rev | 起動形態 |
|---|---|---|---|
| ccbattle bf13fae0 | Complex | 3 | 違反 直列(4msg) |
| mission 8a6e69fa | Critical | 3 | 違反 直列(8msg) |
| navibot 94a9fc52 | Critical | 3 | 違反 直列(5msg) |
| navibot d7e2446d | Standard | 2 | 違反 直列(3msg) |
| smart-social 0f6a7b3a | Complex | 3 | 違反 直列(7msg) |
| smart-social 0deb3b85(worktree) | - | - | 違反 直列(11秒間隔3名) |
| xai-cli c9162ad4 | Complex | 3 | 違反 直列(4msg) |

注: smart-social 0deb3b85 は iter2 scoring で consensus を省略しており Q10 では模範(別軸)。S1(reviewer 起動の並列性)違反と Q10(scorer の M5 遵守)は別観点。

スキル自身が Compact Instructions Top4 に「6ラン中0回しか守られず」と明記し、
「Phase 4 起動時に『今1メッセージでN名出したか?』を自問する」自己チェックを置いている**にもかかわらず、
ドキュメント化後も遵守率 0%**。「自問」型ソフト指示が確率的に機能しないことの実証。
- **対策(ROI 順)**:
  - **(案b) 最優先・即日着手可**: planner 出力に「reviewer N 名を 1 メッセージで起動」を機械生成チェックリスト項目として埋め込み、
    orchestrator がそれをコピペ起動する形にする。SKILL.md / react-loop-details.md への文字列追記で確実に実装可能。
  - **(案a) 要一次検証→格上げ**: PreToolUse hook で reviewer Skill が単独メッセージで来たら警告/ブロック。
    ただし **Claude Code の PreToolUse が `Skill` ツール呼び出しをトリガー/ブロックできるか(toolName 表現)が未検証**。
    一次確認後に採用可否を決める。現状 hooks.json は Stop hook のみ。Codex には効かない点も留意。

### S2. ボトルネックは executor フェーズ(reviewer ではない)
94a9fc52(72min)の内訳: planner 08:27→executor 08:32-09:04(**32min 単一フェーズ**)→reviewer3+scorer 09:04-09:17(13min)。
**executor が wall-clock の支配項**。S1 の並列化も重要だが**速度改善の主戦場は executor**。
- **対策(調査フェーズ込みで分解)**: executor 所要時間の計測フック追加(約1日)→長時間化要因(ツール往復数/再探索)の特定(1-2日)→
  planner SKILL.md の指示密度向上で再探索削減(約1日)。

### S3. orphan run = 62min の作業ロスト + 重複作業

  (b) cross-session 重複警告は **現 cmd_init が issue 番号フィールドを持たない**(mission 文字列のみ)ため前提不成立。
  schema に optional `issue_ref` を追加し init 時に渡す方式が最小コスト。この改修とセットでないと「init 時警告」は実装不能。

### Q1. 同一セッション連続実行で state ロスト 5 件(修正本体は Done・残課題は回帰テスト)
同一 session_id で別ミッションを連続 init すると、修正前のコードでは前ミッションの state.json が
**archive 退避されずに上書き**され score_history が消失。実際に発生した 5 件:

| 消えた mission | session | 上書きした後続 |
|---|---|---|
| smart-social 941476a7 | cc-9d28ff1f | 0f6a7b3a |
| smart-social 6b10d115 | cc-c16a1d32 | ce88bb82 |
| smart-social 243f0183 | cx-019ec9d7 | e5735238 |
| mission 4401d063 | cc-8fe8ee5f | 93aae6fd → 8a6e69fa |
| mission 93aae6fd | cc-8fe8ee5f | 8a6e69fa |

(session cc-8fe8ee5f では 4401d063→93aae6fd→8a6e69fa の **3 ミッションが連鎖上書き**し最後しか残らず)
- **影響**: ①score_history 永久消失(archive scoring .md のみ残存) ②**stats が 5 件 undercount**(可視12 vs 実17)。
- **状況**: 修正(commit 7bb22a9 issue #1-#7 の F2「init archive」)は 14:00 にコード投入済で、
本監査で隔離テストし **現在は機能する**ことを確認(別 mission_id の旧 state を `archive/state-<sid>-<mid>.json` に退避)。
5 件は**修正前(06:33/08:08/12:18/12:53 の上書き)の被害**。
- **残課題(修正本体は Done のため P1 以下)**: (a) 退避 state を stats が拾う回帰テスト(現状 archive/state-*.json は production で未発火=未検証)を CI 化 [P1]
  (b) 既存 5 件を archive scoring .md から最小 state へ復元するワンショット script [P2・任意]。

### Q10. 【最優先】scorer の M5 規律違反 — 差分レビューで consensus に高得点
scoring-rubric.md §119(Issue #4)は「差分レビュー(iter2 以降・検証 1 名)周回では reviewer_consensus に
4.5 以上を付けてはならない(省略が正)」と定める。しかし iter2 差分レビュー **4 run が違反**:

| run | iter2 reviewer_consensus | 判定 |
|---|---|---|
| navibot 94a9fc52 | **5.0(1名固定)** | 明白な違反 |
| xai-cli c9162ad4 | 4.8 | 違反 |
| mission 8a6e69fa | 4.6 | 違反 |
| mission 4401d063 | 4.8 | 違反 |
| smart-social 0f6a7b3a | (consensus 省略) | **準拠(模範)** |

注: ccbattle bf13fae0・smart-social 0deb3b85 は reviewer_count=3 の iter2(複数名)であり M5「検証1名」のスコープ外のため本表から除外(consensus 5.0 でも違反ではない)。

consensus は複数 Reviewer 間の合意度指標であり、1 名固定で高得点を付けると gate の min_item を不当に押し上げる。
4 run はいずれも最終 PASS しており、**M5 違反が甘採点(ungated に近い緩み)の温床**。
- **対策[P1]**: scorer skill に「iter2 差分検証 1 名のときは reviewer_consensus を必ず省略」を機械チェック化
  (push-score 側で「iteration≥2 かつ items に reviewer_consensus 含有 → 警告/reject」)。

### Q2. reviewer_consensus は定義済だが scorer が無視している
当初「定義が無い」と診断したが誤り。**scoring-rubric.md §5(line 99/164)に分散ベースの算出定義が既存**
(「複数レビュアーのスコア分散がどれだけ小さいか」「Scorer が項目5を分散ベースで算出」)。
実態は **定義の不在ではなく scorer がそれを無視して run ごとに別物の値を出している**:
941476a7 iter1=2.0(乖離指標として正しく機能)、94a9fc52 iter1=4.00(分散計測せず固定)、d7e2446d=5.0(合意度)。
Q10 の M5 違反と同根 = **scorer 実装が rubric §5 の分散算出を実装/遵守していない**。
- **対策[P1]**: rubric 追記ではなく、mission-scorer の consensus 算出が分散ベースか検証し、未実装/逸脱なら修正。

### Q3. scorer 出力フォーマットのドリフト
archive scoring .md が run ごとに別形式(フル rubric テーブル / 一行サマリ / 散文)。機械監査・横断比較が困難。
- **対策[P2]**: (a) mission-scorer/SKILL.md に固定テンプレ例(meta + 5項目テーブル + High/Medium/Low件数 + 判定行)を追記=最小コスト。
  (b) --scoring-output の parse 検証スクリプトで強制=工数大。まず (a)。

### Q11. stagnation_count が自動増分されない(停滞停止が実質未実装)
SKILL.md は「過去3イテレーション連続で composite 改善幅<0.1 → 停滞停止」を定めるが、
**mission-state.py の `stagnation_count` は init 時に 0 設定されるのみ(line 343)で自動インクリメントロジックが存在しない**。
11 run 全て `stagnation_count: 0`。今回は全 run が max_iter 3 以内で完了したため露呈しなかったが、
**max_iter=0(上限なし)の長期ミッションでは停滞停止が機能せず無限ループ化リスク**。
- **対策[P2]**: push-score 時に前 iter との composite 差を計算し stagnation_count を自動更新する。

### Q4. 手動スコア補正が不透明
94a9fc52 iter1 で completeness を per-reviewer 平均(3.33)から **3.00 へ「High相当4件補正で足切り」**と主観で下方修正。
判断は妥当だが**再現不能**(補正規則が未明文化)。Q2/Q10 と同根 = rubric の操作化不足。
- **対策[P3]**: 補正規則(High N 件 → 上限 X)を rubric に明記。

### Q5. early-stop の規律は良好(訂正)
- 良: 6b10d115 が iter1 composite4.28 / 残 High=0 で early-stop を明示適用・理由記載。模範。
- 良: xai-cli c9162ad4 は iter1 に **High 2件残存**(scoring .md「Residual High findings: 2 → proceed to iter2」)のため
  iter2 は **early-stop 対象外=規約どおりの必須続行**。当初これを「グレーゾーン続行の記録漏れ」と誤判定したが、
  High 残存が続行根拠であり assumptions への記録も不要。規律違反なし。

### Q6. Complex ミッションのインライン実行(Maker-Checker 回避)
23dd8182(Complex)は reviewer skill を一切呼ばず orchestrator 単独で実装まで完遂しようとした(S3 と同件)。
「Simple はインライン可」ルールの過剰適用 or Maker-Checker スキップ。独立レビューが発生せず(クラッシュで結局未完)。

### Q7. decisions[] が全 run で空

### Q8. codex run の scoring 未永続
e5735238(codex Critical)は archive scoring .md が無い(--scoring-output 未使用)。
codex は --scoring-output の使用が不安定で、スコア根拠が後から検証できない run が出る。

### Q9. assumptions 運用 — session-level は良好・project-level が一部欠落(訂正)
当初「smart-social/ccbattle/navibot は空、mission/xai-cli は充実」と記したが**帰属が逆で誤り**。
一次確認: **session-level assumptions(sessions/<sid>-assumptions.md)は 7 件すべて 11〜35 行の実コンテンツで充実**
(ccbattle 15行 / navibot 3件 11-17行 / smart-social 3件 13-35行)。私が「空」と見たのは **project-level の
.mission-state/assumptions.md**(navibot/smart-social/ccbattle で不在)であり、両者を混同していた。
- **実態**: session-level の Assumption Registry 運用は良好。project-level assumptions.md は 3 プロジェクトで不在だが、
  multi-session 運用では session-level が正なので**実害は小さい**。Finding としては降格(参考情報)。
