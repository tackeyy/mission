## Findings

### P1. `push-score` accepted `iteration=0`, and audit then misreported it as missing iteration 1

Evidence:

- `.mission-state/sessions/cx-019eef1b-d980-7a23-b0fc-d678f8ed0e95.json`
- `mission_id=f9f1c0ee25b4af7d`
- `passes=true`, top-level `iteration=0`, `score_history[0].iteration=0`, composite `4.3`
- `scripts/mission-audit.py` uses `entry.get("iteration") or index`, so `0` is treated as false and reported as missing iteration `1`.
- `skills/mission/bin/mission-state.py cmd_push_score` does not reject `args.iteration < 1`.

Impact:

This creates a pass record outside the normal iter1+ loop and makes the missing-scoring-evidence report point at the wrong iteration. It is not ungated, but it weakens audit precision.

Recommended fix:

- Reject `push-score --iteration 0` with exit 2.
- Add an audit finding for invalid score iteration instead of silently converting 0 to the enumerated index.
- Keep specialist invocations at iteration 0 only if pre-loop evidence is intentionally supported; do not allow scoring at 0.

### P1. Oracle command provider was marked `completed` even though evidence was only "prepared"

Evidence:

- `.mission-state/archive/iter-0-f9f1c0ee-specialist-oracle-reviewer.md`
- `.mission-state/archive/iter-1-f9f1c0ee-specialist-oracle-reviewer.md`
- Both files show `exit_code: 0` and stdout headed `Oracle Browser Review Prepared`, with prompt/result/packet paths and instructions to rerun with `ORACLE_MISSION_WAIT_SECONDS=900`.

Impact:

The state says `oracle-reviewer` completed, but the archive does not contain a review result. That means external command-provider usage was over-counted as applied evidence.

Status:

The later mission `cx-019efccb-1832-7e03-955f-c5d6dadb42d0` says it fixed oracle result-contract handling. This audit confirms the historical defect and recommends a fresh smoke run proving the same prepared-only output is now logged as `prepared`, not `completed`.

### P2. Four historical runs applied specialists after `ask-user` without confirmed selection metadata

Evidence from `unresolved_confirm_specialist_selections`:

- `cx-019eef1b-d980-7a23-b0fc-d678f8ed0e95`: `oracle-reviewer`
- `cx-019ef298-f0d2-7263-8bac-1d2e4475a9bc` archived `23db9575`: `dev-api-designer`, `dev-architect`, `dev-test-strategist`, `development`
- `cx-019ef36a-32e5-7ea1-9951-854a483bdd6d` archived `33943c3f`: `dev-architect`, `dev-code-reviewer`, `dev-test-strategist`, `development`
- `cx-019ef36a-32e5-7ea1-9951-854a483bdd6d` archived `c38d6a7b`: `development`, `strategy-consulting`

Impact:

The actual specialist instructions were used, but the state does not prove that the `ask-user` branch was resolved. This makes it hard to distinguish user-confirmed specialist use from opportunistic inline use.

Recommended fix:

- Keep the newer `--selection-source confirmed-user` path.
- Add a regression fixture for `ask-user` + applied invocation + persisted `specialists_selected`.
- Add audit wording that calls these historical records "applied but unconfirmed", not simply "used".

### P2. One recent self-improvement run had candidate-only specialists

Evidence:

- `.mission-state/archive/worktree-codex-9a52-mission-daily-execution-log/cx-019ef792-f739-7d53-8cb0-b9f40783cfde.json`
- Mission: weekday noon execution-log self-improvement audit since `2026-06-23T03:02:20Z`
- Candidates: `dev-code-reviewer`, `dev-integration-tester`, `dev-unit-tester`, `sc-risk-advisor`
- `specialists_selected=[]`, `specialist_invocations=[]`
- The mission still passed with composite `4.52`.

Impact:

For a log-quality audit that explicitly surfaced testing/risk candidates, no selected/invoked/skipped trail means the run did not actually use external skill evidence and did not even record why it skipped them.

Recommended fix:

- For Standard+ self-improvement/audit missions, require at least a terminal accounting entry for each high-risk or testing/risk candidate.
- If candidates are intentionally not used, log `fallback-core + skipped` with a reason before scoring.

### P2. Blank specialist invocation records exist and can evade accounting

Evidence:

- `.mission-state/archive/state-cx-019efccb-1832-7e03-955f-c5d6dadb42d0-828aedd3.json`
- 6 `specialist_invocations[]` entries have `skill: ""`; the intended skill name is embedded in `role`, e.g. `"architecture-review dev-architect"`.
- `cmd_log_specialist_invocation` only requires the CLI option syntactically; it does not reject an empty string.

Impact:

Blank skill entries can make skip/accounting look present to a human while machine accounting cannot map them to candidates. This is a state quality bug.

Recommended fix:

- Reject empty `--skill` and empty `--role` after trimming.
- Add a mission-audit finding for blank specialist invocation fields.
- Consider a one-time archive repair only if historical stats need to be made exact.
